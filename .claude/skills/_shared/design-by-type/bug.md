# Design by Type: bug（バグ修正）

`type:bug` ラベルが付いた Issue の設計書を書く際の構成指針。

## bug 設計の性格

- 「既存の挙動が壊れている」が前提。**What**（正しい挙動）と **Why**（現状との乖離）をまず固定する。
- 新規 IF 設計は原則なし。既存 IF の振る舞い修正が主。
- 再現条件が曖昧なまま実装に入ると、直したつもりの箇所と真因がずれる → **再現手順を設計書で固定**する。

## 必須セクション

### 1. 概要（1〜2 文）

- どのモジュール / コマンド / API で、**何が** 期待と異なるかを 1 文。
- 例: 「`kaji run` が `--from` 指定時に前段ステップの verdict を読み込まず、強制的に最初から再実行する」

### 2. 背景・目的（OB / EB 必須）

以下 2 セクションを明示的に分けて記述:

- **Observed Behavior (OB)**: 観測された壊れた挙動。ログ / スクリーンショット / 再現したコマンド出力を根拠として引用。
- **Expected Behavior (EB)**: あるべき挙動。根拠（仕様書・既存テスト・ユーザー要求）を 1 次情報で裏付け。

> **Bettenburg et al. (2008)**: 良い bug report は OB + EB + steps-to-reproduce を備える。設計書でもこの 3 点を分離して書く。

### 3. 再現手順（steps-to-reproduce）

最小再現環境と手順を番号付きで列挙:

1. 前提データ / 環境（ワークフロー YAML、環境変数、入力データ）
2. 実行コマンド / 操作手順
3. 観測される出力（OB）

再現できない bug は設計フェーズに入ってはいけない → Issue 本文に再現可能な情報を揃えてから再着手する。

### 4. 根本原因（Root Cause）

コード調査の結果を書く。単なる「ここを直す」ではなく:

- 問題のあるロジックが**なぜ**間違っているか
- **いつから**壊れているか（`git blame` or 関連 PR 番号）
- 同じ原因で**他にも壊れている箇所がないか**の調査結果

### 5. インターフェース

bug 修正では IF は原則維持。変更する場合は:

- 変更前 / 変更後を明示
- 後方互換性の評価（既存呼び出し側への影響）

### 6. 変更スコープ

- 影響するモジュール・ファイル群を明示
- kaji は Python 単一スタック。Scope 分岐は扱わない

### 7. 方針（修正アプローチ）

- 最小侵襲で直す案を優先（影響範囲を局所化）
- リファクタ混在は避ける（混ぜるなら別 Issue 推奨）

### 8. テスト戦略（再現テスト必須）

**bug 固有ルール**: 修正前に Red になる再現テスト（regression test）を必ず 1 本以上定義する。省略不可。

[docs/dev/testing-convention.md](../../../../docs/dev/testing-convention.md) のサイズ定義に従い、根本原因の層に合わせてサイズを選ぶ:

- 純粋ロジック → Small
- ファイル I/O / サブプロセス / 内部サービス結合 → Medium
- 外部 API / E2E → Large

再現テストは OB を assert する形で書き、修正前に FAIL → 修正後に PASS へ遷移することを設計の検証とする。

### 9. 影響ドキュメント

- 仕様書（`docs/cli-guides/`, `docs/reference/python/`, `docs/dev/`）に OB が「正しい挙動」として書かれていた場合、修正対象
- `CHANGELOG` / リリースノートへの記載（運用ルール次第。kaji 現行では未運用）

### 10. 参照情報

- 過去の類似 Issue / PR 番号
- 関連する 1 次情報（仕様書の該当箇所、API リファレンス）

## よくある指摘

- 再現手順が曖昧 → review-design で CR
- 根本原因の調査が浅く「ここを直す」だけで**なぜ**が書かれていない
- 再現テストが設計書に明示されていない
- リファクタが混在している（Scope 逸脱）
