## バグ原因と修正方針レビュー

### Findings

1. **問題 A の validator 方針は追加整理が必要**

`kaji_harness/models.py` の `Step.effort` は `str | None`、`kaji_harness/workflow.py` は未検証で `effort=step_data.get("effort")` を代入し、`kaji_harness/cli.py` は Claude / Codex CLI にそのまま passthrough している。したがって、workflow YAML の `High` / `xHigh` が実行時まで検出されないという原因分析は妥当。

ただし、修正方針の `Literal["none","minimal","low","medium","high","xhigh"]` は Codex の error message 由来であり、Claude 側の `max` を許容するかどうかが未決。Codex と Claude で許容値が完全一致しないなら、global whitelist ではなく以下のどちらかを明記した方がよい。

- kaji workflow として共通 subset のみ許可する
- agent 別 allowed values を workflow parse 時に検証する

`Step` dataclass に `Literal` を付けるだけでは runtime validation にならないため、実装位置は `workflow.py` の YAML load / validation 時点が妥当。

2. **問題 B の修正方針が `comment` に寄りすぎている**

`kaji issue comment` が local issue comment file を作成するだけで commit しない点は事実。ただし未コミット蓄積は comment だけではなく、`kaji issue edit` と `kaji issue close` が更新する `issue.md` も対象になる。

実際に skill 側では `issue-start` / `i-dev-final-check` / `i-doc-final-check` / `issue-fix-ready` などが `kaji issue edit` を使っているため、`kaji issue comment --commit` だけを追加しても `issue.md` 更新が残る可能性がある。

修正方針は `comment` 単体ではなく、local provider の write 系操作 `create/edit/comment/close` のうち、どれを commit 境界に乗せるかとして設計した方がよい。少なくとも今回の運用問題を閉じるには `comment/edit/close` を検討対象に含めるべき。

3. **`issue-close` の自動 commit 救済は条件を強く絞るべき**

`.claude/skills/issue-close/SKILL.md` の base worktree preflight は、未コミット変更があれば ABORT する安全装置になっている。ここを「`.kaji/issues/` 配下だけ自動 commit して続行」に緩和するなら、以下の条件を明記した方がよい。

- dirty file が `.kaji/issues/` 配下のみであること
- local issue 永続化に由来する変更だけを commit 対象にすること
- 自動 commit 後に再度 `git status --porcelain` を確認し、残差があれば ABORT すること

これを曖昧にすると、base worktree の unrelated changes を抱えたまま close が進むリスクがある。

### 推奨修正方針

問題 A は `workflow.py` の parse / validation 時点で `effort` を検証する。値の小文字化だけでなく、大文字が workflow validation で fail する unit test を追加する。許容値は「共通 subset」か「agent 別」かを設計書で決める。

問題 B は `kaji issue comment --commit` だけでなく、local issue write 操作全体の commit 動線として設計する。最小案は `comment/edit/close` に `--commit` を追加し、skill 側の local issue 更新直後に使うこと。加えて `issue-close` は `.kaji/issues/` のみ dirty なら救済 commit、それ以外は ABORT のままにする二段構えが安全。

### 判定

原因分析は大筋で妥当。ただし実装前に以下 2 点を設計条件へ追加することを推奨する。

1. `effort` の許容値を agent 別にするか、kaji 共通 subset にするか
2. commit 動線の対象を `comment` だけでなく `edit/close` まで含めるか
