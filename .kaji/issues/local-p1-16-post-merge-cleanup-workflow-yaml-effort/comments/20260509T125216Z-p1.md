## 対応済み

### Must Fix 1: `_commit_local_issue_change` の atomicity 不成立

**指摘**: `git add <paths> -> git commit -m ...` は index 全体を commit するため、user の他 staged file が HEAD に混入する。temp repo 検証で `target.txt + other.txt` を staged → `git commit -m test` 実行時に両方 HEAD に入ることが確認された。

**修正**:
- helper の実装方針を `git add <paths>` + `git commit -m <msg>` から **`git commit --only/-o -m <msg> -- <pathspec>`** に変更（partial commit）
- Git man page の `--only` 節（"Make a commit by first updating the index with the contents of the paths specified on the command line, ignoring previously staged changes for paths that are not on the command line"）が atomicity を保証
- 仕様: 指定 pathspec の working-tree 内容を **temporary index** に積み直して commit する。事前に staged だった他の file は temporary index に積まれず、HEAD に混入しない。commit 後も元の index にそのまま残る
- 設計書 § インターフェース B の疑似コード書き換え + atomicity の根拠（man git-commit 引用）を明記
- Medium テストに **`test_local_issue_{comment,edit}_with_commit_excludes_unrelated_staged_files`** を追加。事前 staged の `other.txt` が HEAD に混入せず staged のまま残ることを `git show --stat HEAD` + `git diff --cached --name-only` で検証する。**旧設計で FAIL → 新設計で PASS の遷移を再現テストとして記録**

### Must Fix 2: `issue-close` rescue の whitelist が `comments/` 全体を巻き込む

**指摘**: 設計書では「`issue.md` と `comments/*.md` 限定」だが疑似コードは `git add ".kaji/issues/<id>-*/comments/"` でディレクトリ丸ごと add。temp repo 検証で `comments/note.txt` が stage されることが確認された。

**修正**:
- rescue add の glob を **`comments/[0-9][0-9][0-9][0-9]-*.md`** に厳格化（4桁 seq prefix + `.md` 拡張子に限定）
- 条件 1 の dirty path 検査も同じ命名規則 regex で実装し、`note.txt` のような非命名規則ファイルが存在する時点で **ABORT** に倒す
- regex の根拠を一次情報に紐付け:
  - machine_id: `kaji_harness/providers/local.py:36` `_MACHINE_ID_RE = ^[a-z0-9]{1,16}$` （ハイフン無し）
  - slug: `kaji_harness/providers/context.py:17` `_SLUG_RE = ^[a-z0-9][a-z0-9-]{0,39}$`
  - comment file: `kaji_harness/providers/local.py:586` `f"{seq}-{self.machine_id}.md"` (`seq:04d`)
- 確定 regex: `^\.kaji/issues/<issue_id>-[a-z0-9-]+/(issue\.md|comments/[0-9]{4}-[a-z0-9]{1,16}\.md)$`
- rescue commit 自体も `git commit --only -- <pathspec>` の partial commit に統一し、user の他 staged change が HEAD に混入しないことを保証（標準動線と同じ atomicity）
- glob と regex の対応表を § インターフェース B に追加（検査 / add / commit / 再検証の 4 段階で異なる機構を使う理由を明記）
- Medium テストに **`test_issue_close_rescue_excludes_non_markdown_in_comments_dir`** を追加。`comments/note.txt` が rescue commit に巻き込まれず untracked のまま残ることを検証。**旧設計で FAIL → 新設計で PASS の遷移を再現テストとして記録**
- `test_issue_close_rescue_aborts_on_unrelated_dirty_path` / `test_issue_close_rescue_residual_check_after_commit` も追加（条件 1/3 の境界条件保証）
