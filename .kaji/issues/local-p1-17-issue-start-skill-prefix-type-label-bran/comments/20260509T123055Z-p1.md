### 対応済み

- **Finding 1 (High): `labels_to_branch_prefix()` 相当ロジックを skill 側に複製する案は不十分**
  - 採用。修正方針の主軸を **helper CLI 経由** に変更。`kaji issue context <id> --json branch_prefix,branch_name,worktree_dir,branch_prefix_fallback` を `kaji_harness/cli_main.py` に追加し、内部で `provider.resolve_issue_context()` をそのまま呼ぶ薄いラッパーとする。skill 側に label→prefix のロジックは複製しない
  - これにより `kaji_harness/providers/local.py:680` の frontmatter `branch_prefix` 最優先処理を skill 側でも自動的に再現できる
  - Issue 本文「修正方針 / 主軸」「完了条件 / 設計段階」「完了条件 / 実装段階」セクションに反映

- **Finding 2 (High): 明示 `prefix` 引数の扱いが未設計**
  - 採用。**A 案（明示 prefix 引数を廃止、label 一本化）** を確定。判断根拠:
    - `kaji_harness/cli_main.py:1044-1064` `_local_issue_edit` argparse に `--branch-prefix` が無く、明示 prefix を frontmatter に永続化する手段がない（指摘どおり）
    - `grep -rn "branch_prefix:" .kaji/issues/` 実行結果 0 件: 既存 Issue で frontmatter `branch_prefix` を明示設定しているものは存在せず、A 案による regression リスクは無視できる
    - 将来 label と異なる prefix が必要になった時点で、別途 `kaji issue edit --branch-prefix` を追加する形で柔軟性を取り戻せる（後方互換性は確保）
  - Issue 本文「修正方針 / 明示 prefix 引数は廃止（A 案）」「完了条件」両セクションに明記

- **Finding 3 (Medium → Medium-High): 影響度の見積もりが少し低い**
  - 採用。「影響範囲」セクションの深刻度を **medium → medium-high** に格上げ
