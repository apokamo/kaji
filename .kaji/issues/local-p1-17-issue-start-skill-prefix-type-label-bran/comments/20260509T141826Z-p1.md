### 設計の要点

1. **What**: `kaji issue context <id> --json <fields>` helper CLI を新設し、`/issue-start` skill が provider 正本（`provider.resolve_issue_context()`）の `branch_prefix` / `branch_name` / `worktree_dir` をそのまま使う動線に変更する。明示 `prefix` 引数は廃止する（A 案）
2. **Why**: skill 側 default `feat` と provider 側 label-driven mapping が別経路で導出されているため構造的に同期が取れない。skill bash に解決ロジックを複製しても乖離源を残すため、helper CLI で正本を参照する
3. **Constraints**:
   - context 解決ロジックは絶対に重複実装しない（薄いラッパー）
   - frontmatter `branch_prefix` を skill から書ける CLI（`kaji issue edit --branch-prefix`）追加は本 Issue 範囲外
   - `gh issue context` は存在しないため github 経路は dispatcher 内で context を special-case する

### テスト戦略

- **再現テスト（bug 固有・必須）**: `type:bug` Issue で `kaji issue context <id> -q '.branch_prefix'` が `fix` を返すこと
- **Small**: argparse / `--json FIELDS` 絞り込み / `-q` jq 適用
- **Medium**: 全 8 種 type label の parametrized 解決確認、frontmatter override 優先順位、fallback 動作（`type:*` 不在 → `chore`）、github 経路の provider stub
- **Large**: 不要（外部 API / E2E 結合点なし）
