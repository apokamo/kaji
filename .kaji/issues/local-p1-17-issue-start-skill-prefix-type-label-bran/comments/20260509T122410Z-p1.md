レビュー結果: 原因分析は概ね妥当です。ただし修正方針はこのままだと再発余地があります。

**Findings**

1. **High: `labels_to_branch_prefix()` 相当ロジックを skill 側に複製する案は不十分です。**  
   context の正本は label だけではなく、local では `branch_prefix` frontmatter を label より優先します。根拠は `kaji_harness/providers/local.py:680` の `meta.get("branch_prefix")` 優先処理です。一方、Issue の修正方針は `.kaji/issues/local-pc5090-17-issue-start-skill-prefix-type-label-bran/issue.md:69` で「labels を取得して相当ロジック」としており、frontmatter override を再現できません。  
   修正は `kaji_harness` 側に `resolve_issue_context()` を使う helper CLI を追加し、skill はそれを読む方針に寄せるべきです。

2. **High: 明示 `prefix` 引数の扱いが未設計です。**  
   `/issue-start <issue_id> fix` のように明示 prefix を指定した場合、現在の skill は worktree/body NOTE だけを更新し、local frontmatter の `branch_prefix` は更新しません。`.claude/skills/issue-start/SKILL.md:39` から `.claude/skills/issue-start/SKILL.md:89` までを見る限り、context 正本への永続化がありません。さらに `kaji issue edit` も `branch_prefix` 更新を受け付けません `kaji_harness/cli_main.py:1044`。  
   つまり「デフォルト prefix」だけ直しても、明示 prefix で同じ乖離が残ります。方針としては、明示 prefix を廃止/非推奨にして type label に寄せるか、明示 prefix を provider 側に永続化する手段を作る必要があります。

3. **Medium: 影響度は少し低く見積もられています。**  
   Issue では「fallback 解釈で吸収」とありますが、skill 定義上は `[branch_name]` をそのまま merge/PR 操作に使う箇所があります。例: local close の `git merge --no-ff --no-edit [branch_name]` `.claude/skills/issue-close/SKILL.md:280`。agent が毎回賢く Issue 本文側へ fallback する保証はないので、medium ではなく medium-high 寄りです。

**推奨方針**

`kaji issue context <issue_id> --json branch_prefix,branch_name,worktree_dir,branch_prefix_fallback` のような helper CLI を追加し、内部で既存の `provider.resolve_issue_context()` を呼ぶのが一番堅いです。`issue-start` は prefix 省略時にその helper の結果を使う。明示 prefix は、context と一致しない場合に ABORT するか、provider-aware に正本へ永続化する仕様を追加してください。

テスト観点は、8 種 label と fallback だけでなく、`branch_prefix` frontmatter 優先、明示 prefix 不一致時の挙動、既存 `feat/<id>` 運用中 Issue の扱いまで入れるべきです。
