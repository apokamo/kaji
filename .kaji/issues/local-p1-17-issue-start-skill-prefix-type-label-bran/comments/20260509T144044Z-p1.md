## 指摘事項 (Must Fix)

- [ ] **High** [`kaji_harness/cli_main.py:995`](</home/aki/dev/kaji/kaji-fix-local-pc5090-17/kaji_harness/cli_main.py:995>): `kaji issue context` が GitHub provider の失敗を捕捉できていません。`_handle_issue_context()` が握っているのは `IssueNotFoundError` / `LocalProviderError` / `ValueError` だけですが、GitHub 経路の `resolve_issue_context()` は `GitHubProviderError` を投げます（[`kaji_harness/providers/github.py:56`](</home/aki/dev/kaji/kaji-fix-local-pc5090-17/kaji_harness/providers/github.py:56>)-86）。実際に `GitHubProvider.resolve_issue_context` を `GitHubProviderError('boom')` で patch すると、`_handle_issue(['context', '123'])` は exit code を返さず例外がそのまま外へ漏れました。設計書と docs は `provider.type='github'` をサポート対象としているので、この経路は `stderr` 整形 + 安定した exit code に落とす必要があります。
- [ ] **Medium** [`.claude/skills/issue-start/SKILL.md:50`](</home/aki/dev/kaji/kaji-fix-local-pc5090-17/.claude/skills/issue-start/SKILL.md:50>): `/issue-start` が `kaji issue context` の出力を読むために system `jq` を必須化しています。`kaji_harness/cli_main.py:844-894` は Python `jq` package 経由の `-q` を既に提供しており、同ファイル `:861-863` にも shell 変数代入向け raw 出力契約が明記されています。ここで外部 `jq` を呼ぶと、`jq` バイナリが無い環境で `/issue-start` だけが失敗し、CLI 側で撤去した依存を skill 側で再導入する形になります。`kaji issue context ... -q '.branch_prefix'` 等で jq-less に統一してください。
