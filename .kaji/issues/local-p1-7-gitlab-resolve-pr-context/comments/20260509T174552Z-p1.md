### 設計の要点

1. **What**: `GitLabProvider.resolve_pr_context(branch_name) -> PRContext | None` を追加し、`prompt.py` / `runner.py` 経路で `pr_id` / `pr_ref` を自動注入する。新 DTO `PRContext(pr_id, pr_ref)` を導入。
2. **Why**: bucket `local-pc5090-1` Phase 4 申し送りの GitLab 先取り実装。skill 側の暫定運用記述（`kaji pr list --search`）を将来削除可能にする前提を作る。
3. **Constraints**:
   - `prompt.py` / `runner.py` に `provider.type == "gitlab"` 等の分岐を入れない（Issue 完了条件）
   - 既存 `resolve_mr_iid_from_branch` を再利用し subprocess hit を二重化しない
   - `IssueContext.branch_name` を一次情報とし、runner cwd の `git rev-parse` に依存しない
   - GitHub provider 側の `resolve_pr_context` 実装は本 Issue OUT スコープ

### 主要な設計判断

- **`PRContext` を `IssueContext` と分離**: PR は workflow 実行中に新規作成されるため `IssueContext` 解決時点で確定できない。`prompt.py` 内で conditional マージ。
- **`runtime_checkable` Protocol への影響回避**: GitHub / Local provider が Protocol 不充足になる副作用を避けるため、本 Issue では Protocol 変更を行わず `prompt.py` / `runner.py` 側で `getattr` ガード。
- **PR 解決失敗の挙動**: 該当 0 件 → `None`、API エラー → WARN + `None`（workflow 全体を止めない判断）
