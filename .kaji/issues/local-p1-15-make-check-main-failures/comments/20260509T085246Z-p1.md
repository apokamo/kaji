### 設計の要点

1. **What**: `make check` を main 直で clean 通過させる（テスト 10 件 fail 解消）
2. **Why**:
   - 原因 A: `_handle_pr` 直呼びテスト 9 件が `_load_config_for_dispatch` の CWD 依存を mock していないため、`.kaji/config.local.toml`（gitignored、`type=local`）配下で bare-provider gate に early-exit する
   - 原因 B: Issue 5 commit `a60e6f4` で `gitlab` が valid 値に昇格したが `tests/test_phase4_large_local.py:188` の `requires_provider: gitlab` fixture 更新が漏れた
3. **Constraints**:
   - production 側（`kaji_harness/cli_main.py` `_handle_pr` / `_load_config_for_dispatch`）は変更しない（Phase 4 fail-fast 契約は維持）
   - 修正は `tests/` 配下のみ
   - `tests/conftest.py` への autouse 追加はしない（`test_phase3c_dispatcher.py` 系 / `test_existing_pr_view_fails_when_config_missing` との衝突を避けるため class-scoped に局所化）

### テスト戦略

- 既存 9+1 テストを「修正前 Red / 修正後 Green」の再現テストとして使う（追加テストは作らず）
- 既存 passing test `test_existing_pr_view_fails_when_config_missing` が autouse 追加後も継続 PASS することを確認（pytest patch stack で内部 `with patch` が autouse `monkeypatch.setattr` を上書きする仕様）
- main 直 / worktree 配下の両環境で `make check` が clean 通過することを実装段階で検証

### 主な設計判断

- **採用**: 各 affected class に `@pytest.fixture(autouse=True)` で `_load_config_for_dispatch` を stub する方式
- **不採用候補**: (a) `monkeypatch.chdir(tmp_repo)` + 一時 config 書き出し（`test_phase3c_dispatcher.py` 系のパターン）→ argv 組み立て検証が主旨のテストに対しては overkill。(b) `tests/conftest.py` global autouse → 既存 isolation pattern との衝突リスク
- **採用根拠**: 同 file の passing test (`test_existing_pr_view_fails_when_config_missing`) が同一 patch target を既に使用しており、横展開がパターン継承で完結する
