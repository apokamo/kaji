## 概要

設計の主方向は妥当です。`IssueContext` と `PRContext` を分離し、branch から PR/MR を逆引きして prompt に conditional 注入する方針は、bucket 申し送りと EPIC の「skill 側に GitHub/GitLab 分岐を入れない」という原則に整合しています。

一方で、現状の設計書には「この Issue で何が完了するのか」が 2 箇所で食い違っています。特に `provider.type='gitlab'` で `/pr-fix` と `/pr-verify` がそのまま使える前提は、現行 skill の Step 0 ガードと矛盾しています。このまま実装に進むと、prompt 注入経路ができても対象 skill 自体が起動せず、ユーザーストーリーを満たしたことになりません。

## 改善提案 (Should Fix)

- **`_resolve_pr_context_safe` の例外吸収範囲を絞ってください**
  提案 helper は `except Exception` で全例外を `WARN + None` に落としています。`docs/reference/python/error-handling.md` は広すぎる catch を禁止しており、これだと契約逸脱や単純な実装バグまで静かに隠れます。想定しているのが provider 外部依存の失敗だけなら、少なくとも provider 由来の既知例外に限定するか、「契約違反は落とす / 外部失敗だけ degrade する」の線引きを設計に入れてください。

- **`pr_context` を「keyword-only」と書くなら、Python の signature もその前提で統一してください**
  設計書では keyword-only と説明していますが、提示コード断片には `*` がなく通常の optional positional 引数に見えます。大きな設計問題ではありませんが、backward compatibility の説明とコード例は揃えておくべきです。
