## 指摘事項 (Must Fix)

- [ ] **`kaji_harness/providers/gitlab.py:60-88,185-198,250-286`**: `gitlab.com` 固定が未実装です。Issue 本文は `hostname` フィールドを持たず `gitlab.com` を内部固定すると明記し、さらに `glab` の current project / login に依存しないことを要求しています（`.kaji/issues/local-pc5090-5-gitlab-provider-impl/issue.md:26-28,37-40`）。しかし実装は `glab api` を `glab api <endpoint>` のまま呼び出しており、`glab api --help` では「current directory が Git directory の場合はその authenticated host を使い、`--hostname` で override する」と明記されています。また mutating 系も `--repo group/project` の bare path を渡すだけで、`glab issue create --help` が示す full URL / git URL の強制指定を使っていません。これだと self-hosted GitLab を使う環境では別 host に飛ぶ、または期待と異なる host 解決になります。`glab api` 側は `--hostname gitlab.com` を明示し、mutating 系は `--repo https://gitlab.com/<group>/<project>` のように host を固定できる形へ変更し、その分の回帰テストも追加してください。

## 改善提案 (Should Fix)

- **`kaji_harness/providers/gitlab.py:225-226,303-319`**: `view_issue()` / `list_issues()` / `list_labels()` は 1 ページ目しか取得していません。`glab api --help` には `--paginate` があり、現状だと comment / issue / label が 100 件を超えたときに情報が欠落します。少なくとも `view_issue()` の notes は workflow の証跡取得に直結するので、pagination 方針を実装か明文化のどちらかに寄せた方が安全です。
