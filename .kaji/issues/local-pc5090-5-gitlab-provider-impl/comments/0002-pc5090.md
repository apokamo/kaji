### 設計の要点

1. **What**: `IssueProvider` Protocol 8 メソッドを `glab` CLI subprocess + `glab api` REST のハイブリッドで実装し、`provider.type='gitlab'` 配下で `get_provider()` が `GitLabProvider` を返す状態にする
2. **Why**:
   - 確定事項 #1（`glab` CLI 必須化）+ #3（gitlab.com 固定 / `gl:N` 規約）+ #5（本格実装）に従い、`GitHubProvider` と対称構造で実装することで保守コストを最小化
   - `glab` CLI は構造化 JSON 出力を持たない（`glab issue list --output-format` の値域は `details/ids/urls`）ため、read 系は `glab api projects/:id/...` 経由で REST JSON を取得
3. **Constraints**:
   - mutating 系（create/edit/comment/close）は `glab issue <sub> --repo <repo>`、read 系（view/list/list_labels）は `glab api projects/<urlencoded-repo>/...` 経由
   - `--repo` は context 暗黙依存禁止（明示指定必須）
   - `[provider.gitlab]` は `repo`（必須）+ `default_branch`（既定 `"main"`）のみ。`hostname` は持たない（gitlab.com 固定の論理的帰結）
   - `gl:N` ID 拡張 / `kaji issue` passthrough / `kaji pr` MR エイリアス / cache populate は子 Issue #2 以降のスコープ。本 Issue では `view_issue` 等は数値文字列 IID をそのまま受け取る
