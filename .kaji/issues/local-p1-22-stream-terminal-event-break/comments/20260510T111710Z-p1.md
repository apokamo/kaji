## 概要

Claude の観測事実に基づく root cause と再現戦略は妥当です。ただし、現状の設計のままでは「terminal event を見たのに timeout 扱いになる競合」と「Codex の失敗終端を見逃して同種ハングを残す穴」があるため、実装着手はまだ早いです。

## 指摘事項 (Must Fix)

- [ ] **terminal event 後の grace wait 中に既存 timer が発火する競合を潰してください**
  - 設計書は `stream_and_log()` で terminal event 検知後に `process.terminate() -> wait(5) -> kill()` するとしていますが、一次情報の現行実装 `kaji_harness/cli.py:117-131` では timeout timer は `_execute_cli_once()` の finally でしか `cancel()` されません。
  - このまま `stream_and_log()` 側で 5 秒の grace wait を持つと、terminal event が timeout 直前に届いたケースで timer がその最中に発火し、`timed_out` が立って `StepTimeoutError` になります。つまり「論理終端は見えているのに timeout 扱い」という新しい race を入れます。
  - **必要な修正**: terminal event 観測時点で timeout をどう disarm するかを設計に明記してください。例えば `stream_and_log()` が `terminal_seen` を返し、`_execute_cli_once()` 側で timer を先に止めてから後始末する、あるいは post-terminal cleanup 用の別 budget を持つ、のいずれかが必要です。

- [ ] **Codex の terminal event を「暫定」で済ませず、成功/失敗の終端行列を一次情報で確定してください**
  - 設計書は `CodexAdapter` を `event.get("type") == "turn.completed"` としていますが、「実装時に再確認」と保留しています。review-design の段階でここを未確定のままにすると、interface 追加の根拠が不足しています。
  - しかも一次情報 `docs/cli-guides/codex-cli-session-guide.md:243-252` には `turn.completed` のほか `turn.failed` と `error` が明記されています。既存コードも `tests/test_codex_robustness.py:272-320` で `turn.failed` / `error` を failure path として扱っています。
  - 現設計の `is_terminal_event()` が `turn.completed` のみだと、Codex の失敗ケースでは依然として fd leak 時に timeout までぶら下がります。bug 修正としては片手落ちです。
  - **必要な修正**: adapter ごとに「success terminal / failure terminal / 非 terminal」を一次情報つきで表にし、Codex で `turn.failed` をどう扱うかを明記してください。もし一次情報が不足するなら、今回のスコープを Claude のみへ絞り、Codex/Gemini は follow-up Issue に切り出す方が筋が通ります。
