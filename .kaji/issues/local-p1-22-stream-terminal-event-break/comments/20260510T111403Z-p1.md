### 設計の要点

1. **What**: `CLIEventAdapter` に `is_terminal_event(event) -> bool` を追加し、`stream_and_log` で受信時に for loop を break + `terminate() → wait(5) → kill()` で後始末する。
2. **Why**: stream-json の「論理的 session 終了（terminal event）」と「stdout fd の物理的 close」が独立している。現状実装は後者だけに依存しているため、子孫プロセスの fd leak で `default_timeout` まで無駄に block する。
3. **Constraints**:
   - 既存 `CLIResult` 構造は不変（後方互換）
   - timer (`_kill_process`) は最終ガードとして温存
   - terminal event を出さない CLI / クラッシュ経路は従来通り EOF or timer で守る
   - リファクタは混ぜない（`stream_and_log` の責務再設計は別 Issue）

### テスト戦略

- **再現テスト（Medium、必須）**: terminal event を出した後 stdout を close せず sleep する fake CLI を `default_timeout=3` で実行。修正前は `StepTimeoutError`、修正後は 3s 以内に成功完了。
- **Small**: 各 adapter の `is_terminal_event` 判定（`type == "result"` 等）。
- **Medium regression**: 正常 EOF 経路、terminal event なし経路、timer 経路（最終ガード）が従来通り動作することを確認。
- **Large**: 不要。Medium の fake subprocess でロジックを完全カバー、Large 追加で得られる回帰検出情報がほぼないため（testing-convention 4 条件の 1, 3 を充足）。
