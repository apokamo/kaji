## 指摘事項 (Must Fix)

- [ ] **`kaji_harness/cli.py:154-159`**: `terminal_seen=True` 時の失敗判定を `result.error_messages` の有無だけに寄せたため、Claude / Gemini の terminal error が成功扱いになります。`ClaudeAdapter.is_terminal_event()` は `type == "result"` を無条件に terminal とし（`kaji_harness/adapters.py:107-108`）、`GeminiAdapter.is_terminal_event()` も `type == "result"` を無条件に terminal としています（`kaji_harness/adapters.py:181-182`）。さらに追加テスト自身が Claude `subtype:"error"` と Gemini `status:"error"` を terminal と定義しています（`tests/test_adapters.py:485-488`, `tests/test_adapters.py:530-533`）。それにもかかわらず `_execute_cli_once()` は terminal 経路で `process.returncode` や result event 自体の failure 情報を見ず、`error_messages` が空ならそのまま成功 return します。実際に以下を独立再現しました。
  - Claude 再現: `{"type":"result","subtype":"error","is_error":true}` を出して `exit 1` する mock CLI を `execute_cli()` に流すと、`CLIExecutionError` ではなく `CLIResult(terminal_seen=True, stderr='', error_messages=[])` が返る。
  - Gemini 再現: `{"type":"result","status":"error","stats":{}}` を出して `exit 1` する mock CLI でも同様に成功 return する。
  - Claude の result は仕様上 `subtype` が `success / error` を持ち、`is_error` フラグもあるため（`docs/cli-guides/claude-code-cli-guide.md:368-370`）、ここを無視してはいけません。Gemini も `result` が最終結果イベントであり（`docs/cli-guides/gemini-cli-session-guide.md:623-631`）、failure 情報を terminal result から判定できる前提です。
  - 必要な修正は、terminal event 後の失敗判定を `error_messages` 依存にせず、少なくとも `process.returncode != 0` と adapter/イベント固有の failure シグナル（Claude `is_error` / `subtype:error`、Gemini `status:error`、Codex `turn.failed`）を整合的に扱うことです。あわせて Claude/Gemini failure terminal の回帰テストを追加してください。

## 改善提案 (Should Fix)

- **`tests/test_cli_streaming_integration.py`**: 現在の Medium テストは Claude/Gemini の success terminal と Codex の failure terminal に偏っています。今回の回帰を防ぐには、Claude `result(error)` と Gemini `result(status=error)` が `CLIExecutionError` になることを直接固定した方がよいです。

- `timer.cancel()` を terminal 観測直後に寄せた設計と `timed_out.is_set() and not result.terminal_seen` のガードは、前回設計レビューで指摘された race を素直に閉じています。
