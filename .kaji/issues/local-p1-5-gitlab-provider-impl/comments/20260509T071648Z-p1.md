## 概要

GitLab provider を `glab` subprocess + `glab api` のハイブリッドで実装する主方針自体は妥当です。`iid` 採用、`opened`→`open` 正規化、`system` note 除外、`resolve_issue_context` の組み立て方も既存構造と噛み合っています。

ただし、現状の設計は **Issue 本文で明示された要求とズレている点** と、**変更後に stale になる docs / help の影響評価が不足している点** があるため、このまま実装着手は避けるべきです。

## 指摘事項 (Must Fix)

- [ ] **`ProviderConfig.type` の型契約が Issue 要求から後退している**
  - Issue 本文は `ProviderConfig.type Literal に "gitlab" 追加` を IN scope として明記しています（`.kaji/issues/local-pc5090-5-gitlab-provider-impl/issue.md:38-41`）。
  - しかし設計書では `ProviderConfig.type` を `str` のまま据え置いています（`draft/design/issue-local-pc5090-5-gitlab-provider-impl.md:121-128`）。
  - `Workflow.requires_provider` では `Literal` を採用する設計にしている一方（同:170-174）、provider 側だけ `str` に戻す理由が書かれていません。これは Issue の要求未充足であり、型安全性の観点でも後退です。
  - **修正方針**: `ProviderConfig.type` を `Literal["github", "local", "gitlab"]` にするか、どうしても `str` を維持するならその理由と影響を Issue 要求の変更として明示してください。

- [ ] **影響ドキュメント評価が不足しており、実装後に docs / help が stale になる**

  - それにもかかわらず設計書の影響ドキュメント表は `workflow-authoring.md` を候補に挙げておらず、外部 docs 更新を子 Issue #5 に送っています（`draft/design/...md:426-439`）。
  - `docs/dev/documentation_update_criteria.md:15-35` でも、CLI / workflow / dev docs の変更は影響評価対象です。今回の変更は「実装内部のみの軽微修正」ではありません。
  - **修正方針**: 少なくとも `docs/dev/workflow-authoring.md` と、provider 値域を説明する help/docstring 群を影響対象に追加し、「本 Issue で更新するもの」と「子 Issue へ送れるもの」を切り分けてください。

## 改善提案 (Should Fix)

- **`list_labels()` の pagination 方針を明記する**
  - 設計本文は `glab api projects/<repo>/labels?per_page=200` を想定しています（`draft/design/...md:66`）。
  - 一方で設計書自身の一次情報欄は GitLab REST の `per_page` 上限が 100 だと書いています（同:447）。
  - 本 project のラベル数が 100 未満でも、一次情報と矛盾したままにしない方がよいです。`per_page=100` に修正するか、必要なら paginate する方針を書いてください。

- **`create_issue()` が `Issue` を返すための取得経路を 1 行でも補う**
  - mutating 系を `glab issue create` に寄せる方針は理解できますが、作成直後にどう IID を得て `Issue` を返すかが table 上では未記載です。
  - stdout から URL/IID を抽出して `view_issue()` するのか、作成だけ `glab api` に寄せるのかを明記しておくと、実装時のブレが減ります。
