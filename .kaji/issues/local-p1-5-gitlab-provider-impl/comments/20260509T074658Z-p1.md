### Must Fix #1 への対応: gitlab.com host pinning

**実装方針**: レビュー suggest（mutating 系で full URL `--repo https://gitlab.com/...`）ではなく、`glab` の global flag `--hostname gitlab.com` を `_run_glab` で全 invocation に default 注入する形を採用しました。

**根拠（反論ではなく技術的妥当性に基づく選択）**:

1. `glab --help` の global flags に `--hostname` が定義されており、`glab api` / `glab issue *` の双方に対し host 解決を強制 override する。これにより `glab api <endpoint>` も `glab issue create --repo group/project` も同一の機構で host が固定される
2. mutating 系で full URL を渡す案は二重防衛的だが冗長で、`--hostname` だけで host 固定の保証としては十分（GitLab REST の `:id` は URL-encoded path / numeric ID を受理するため、`api` 側は URL 埋め込みで完結）
3. 全 invocation で同じ前置を共有することで、後続子 Issue (#2 の MR 経路、#5 の docs、#6 の E2E) でも host pinning ロジックを 1 箇所（`_run_glab`）で集中管理できる

**実装**:

```python
# kaji_harness/providers/gitlab.py
_GITLAB_HOSTNAME = "gitlab.com"

def _run_glab(self, *args, capture=True):
    ...
    cmd = ["glab", "--hostname", _GITLAB_HOSTNAME, *args]
```

**回帰テスト**: `TestHostnamePinning` クラスを新設、view_issue / list_issues / list_labels / create_issue / edit_issue / comment_issue / close_issue の **7 method 全てで `--hostname gitlab.com` が cmd 先頭に挿入されている**ことを assert（既存テストの index ずれも修正）。
