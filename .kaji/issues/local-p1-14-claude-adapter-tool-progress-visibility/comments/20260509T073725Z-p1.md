### 設計の要点

1. **What**: `ClaudeAdapter.extract_text` を改修し (A) `assistant` の `tool_use` / `thinking` ブロックも 1 行サマリで抽出、(B) `result` イベントからのテキスト抽出を廃止。`extract_cost` は `result` を引き続き処理
2. **Why**: 現状コードは (A) `text` ブロックしか抽出せず `tool_use` 50 件 / `thinking` 6 件が `stream_and_log` に届かない（異常 A）、(B) `assistant` 最終 text と `result.result` を二重抽出して同一テキストが 2 回 print される（異常 B）。Codex / Gemini アダプタが取っている対称設計（テキスト抽出は assistant 系のみ、result/turn.completed は cost のみ）に合わせる
3. **Constraints**:
   - `CLIEventAdapter.extract_text` のシグネチャ（`event -> str | None`）は維持
   - `extract_cost` の挙動は不変
   - tool 入力の summary 切り詰め長は **80 文字**（誤マッチ防止 + 進捗視認性のバランス）
   - 未知ツールはツール名のみ（`input` repr は出さない: 秘匿情報混入リスク回避）

### verdict parser への影響評価

`full_output` に `[tool] ...` 行が混入する影響を verdict.py の 3 段パーサに対して評価:

| パース段 | 誤マッチリスク |
|---|---|
| Step 1 (Strict `---VERDICT---`) | なし |
| Step 2a (Relaxed delim) | なし |
| Step 2b (Key-value `status: PASS` 等) | 理論上ありうるが、Step 1/2a 通過後しか到達せず、かつ valid_statuses 完全マッチ要件で実質発生せず |

→ **追加の防御コードは入れない**（YAGNI）。観測されたら別 Issue で対処。
