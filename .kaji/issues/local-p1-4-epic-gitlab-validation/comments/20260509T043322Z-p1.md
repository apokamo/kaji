## OQ-2 レビュー結果

### 概要

OQ-2 の決定方針「(d) skill が使う subcommand に限定 + kaji 側 contract に揃える」は妥当です。silent passthrough を禁止し、skill 側に GitHub/GitLab 分岐を入れない方針も、既存 provider 抽象の方向性と整合しています。

ただし、`draft/lab/gitlab-validation/kaji-pr-mr-bridge.md` の contract としては、以下 2 点がまだ実装時に割れる可能性があります。

### 指摘事項 (Must Fix)

1. **`kaji pr review --request-changes --body-file` を `glab mr revoke` 単体に対応させるのは不十分**
   - 既存 `pr-verify` は `kaji pr review [pr_id] --request-changes --body-file -` で、本文つきの正式なレビュー差し戻しを投稿します。
   - 一方、`glab mr revoke` は approval を取り消すだけで、body を投稿する contract ではありません。
   - GitLab 側では少なくとも `glab mr note --message <body>` と `glab mr revoke` の組み合わせ、または「request-changes は note 投稿 + 非承認状態の表現」として provider contract を明文化してください。
   - `--approve --body-file` についても、`approve` が body を受けないなら body を note として残すか、body を捨てるのかを明記してください。skill contract 的には body を捨てない方が妥当です。

2. **`kaji pr merge` の no-ff / squash 禁止契約が `glab mr merge` でどう担保されるか未定義**
   - 現行 `kaji pr merge` は GitHub 側で method flag を剥がして `--merge` を強制し、kaji の `--no-ff` 運用に寄せています。
   - `glab mr merge` は `--squash` / `--rebase` は持ちますが、GitHub CLI の `--merge` に相当する明示 flag はなく、実際の merge method は GitLab project 設定に依存し得ます。
   - OQ-2 文書の `merge: ほぼ素通し` は弱いです。少なくとも「squash/rebase 相当の user flag は露出しない」「GitLab project の merge method / squash default を `gitlab-mode.md` または preflight で確認する」「kaji が保証できる範囲と project 設定に委ねる範囲」を明記してください。

### 判定

Changes Requested — OQ-2 の採用方針自体は承認できますが、上記 2 点を `kaji-pr-mr-bridge.md` に追記してから子 Issue 起票に進むのが安全です。
