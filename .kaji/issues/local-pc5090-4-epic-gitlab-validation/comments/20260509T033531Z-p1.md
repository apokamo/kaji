## 概要

EPIC の方針（GitLab 採用を確定せず、採用可能な準備状態を先回りで作る）は妥当です。tracking Issue と子 Issue の責務分離も、前回レディネスレビュー後の本文では整理されています。

ただし、現行実装上 `gitlab` provider は config / workflow schema の enum に存在せず、GitLab の ID 体系も `id` と `iid` が分かれます。この2点を EPIC の子 Issue 構成に明示しないまま進めると、子 Issue #1〜#2 の実装で設計前提が割れます。

## 指摘事項 (Must Fix)

- [ ] **provider type / workflow schema 拡張が子 Issue 構成から漏れている**: 完了条件に `provider.type='gitlab'` で workflow 完走がある一方、現行 `config.py` は `provider.type` に `gitlab` を許容せず、`workflow.py` / `models.py` も `requires_provider: gitlab` を許容しません。子 Issue #1 または #2 に、少なくとも以下を明記してください。

  - `[provider.gitlab]` の config dataclass / validation / overlay merge
  - `get_provider()` の `GitLabProvider` 分岐
  - `Workflow.requires_provider` enum への `gitlab` 追加
  - builtin workflow の `requires_provider` 設計（github/gitlab/local/any のどれにするか）
  - `kaji validate` / workflow provider match のテスト更新

- [ ] **`gl:N` の ID 契約が GitLab の `id` / `iid` 差分に対して未定義**: GitLab API は project の `id` と issue の `issue_iid`、MR の `merge_request_iid` を分けます。本文の `gl:N` が「GitLab Issue の project-local IID」を指すのか、「global issue id」を指すのかが未定義です。`normalize_id` と `kaji issue view gl:N` の契約として、`N` の意味、対応する configured project、表示用 `issue_ref`、MR 側の `pr_id` / `pr_ref` との関係を子 Issue #2 に明記してください。

- [ ] **`provider.gitlab` の最小 config 契約が未定義**: GitHubProvider は `provider.github.repo` を `--repo` に渡して cwd / git remote 推論に依存しない構造です。一方、`glab auth status` は current context / `GITLAB_HOST` / config から対象を解決し得ます。kaji の provider は既存設計に合わせ、`provider.gitlab.repo = "group/project"`、`default_branch`、必要なら `hostname = "gitlab.com"` のような明示 config に寄せるのか、`glab` context に任せるのかを子 Issue #1 の設計前提として固定してください。self-hosted 非対応を維持するなら、`hostname` を持たない/`gitlab.com` 固定にする理由も明記が必要です。

- [ ] **OQ-2 が子 Issue 起票前必須なのに、完了条件側で保護されていない**: `kaji pr` MR エイリアス範囲は子 Issue #2 のスコープに直結します。本文では「子 Issue 起票前に確定必須」とありますが、完了条件には `decision-notes.md` 作成や OQ-2 決着が含まれていません。子 Issue 8本を起票する前に OQ-2 を決めることを、完了条件または進め方のゲートとして明示してください。

## 改善提案 (Should Fix)

- **GitLab MR review-comments の扱いを子 Issue #6 だけに閉じず、#2/#3 の interface 契約にも反映**: GitLab の MR discussion API は thread / note / position / resolved を持ち、GitHub PR review comment と完全同型ではありません。`kaji pr review-comments` / `reply-to-comment` 相当を「互換 subset」にするのか、「GitLab 専用 shape を provider で GitHub 互換に変換する」のかを早めに決めると、#6 の E2E で手戻りが減ります。
