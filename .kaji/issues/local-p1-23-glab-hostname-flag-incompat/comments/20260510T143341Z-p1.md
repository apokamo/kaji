### 設計の要点

1. **What**: `glab` 起動 3 サイト（providers/gitlab.py:_run_glab、cli_main.py:_forward_to_glab、sync.py:_glab_api_get）から `--hostname <host>` の引数注入を撤去し、`subprocess.run(env={**os.environ, "GITLAB_HOST": "gitlab.com"})` の env 注入に統一。3 サイト共通の `_glab_env()` helper を導入する。
2. **Why**: `glab` v1.36/1.95 双方で `--hostname` は global flag ではなく `api` / `auth` 系 sub-flag。`issue` / `mr` 系 subcommand は parser が知らず Unknown flag で reject する。`glab --help` / `glab api --help` 双方で hostname 切替の正規ルートは `GITLAB_HOST` env と明記。
3. **Constraints**: 公開 IF（CLI / Provider Python API）不変。`docs/cli-guides/gitlab-mode.md` の `glab auth login --hostname` 記述は `auth login` の正規 sub-flag のため不変。large_gitlab tests の `glab api --hostname` 呼び出しは正規 sub-flag 用法のため本 fix スコープ外。

### 偶発成功の説明（設計書 § 根本原因）

実機再確認: `glab --hostname X api user` → 200 OK / `glab --hostname X issue list` → Unknown flag。`api` subcommand に到達したケースのみ flag parser が `--hostname` を sub-flag として吸収するため、`sync._glab_api_get` と provider の read 系 (`glab api` 経路) のみ偶発的に動作していた。`issue` / `mr` 系 mutating は全 reject。

### テスト戦略

- **Small（再現テスト・必須）**: 3 サイトすべてで `subprocess.run` の `cmd` に `--hostname` が **含まれない** こと、`env` kwarg に `GITLAB_HOST="gitlab.com"` が **含まれる** ことを assert。修正前 FAIL → 修正後 PASS。
- **Medium**: 不要（subprocess args/env 変更のみ。Small で観測完結）。
