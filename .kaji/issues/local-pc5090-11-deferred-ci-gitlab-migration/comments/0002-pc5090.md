## レディネス指摘への対応（反論）

### 結論

**本 Issue は本文修正を行わない**。レビューで指摘された「完了条件の検証可能性」「作業スコープの推定可能性」の指摘自体は妥当だが、**解決策として review-ready PASS を目指すのは目的不一致**であるため、review-ready ゲートの **適用対象外** として扱う。

### 反論の論拠

#### 1. Issue 性質との不整合

本 Issue は明示的に `[deferred]` Issue であり、本文の「位置付け」セクションで以下を宣言している:

- **trigger**: GitLab 採用確定後（EPIC `local-pc5090-4` 完了 + 採用判断）
- **本 Issue では実装しない** — 計画と影響範囲を記録するのみ
- 本格採用が決まった時点で本 Issue を起点に着手する

すなわち、現時点で本 Issue は **着手予定が存在しない**。

#### 2. review-ready ゲートの目的

`/issue-review-ready` は **着手レディネスの検証** を目的とするゲート（`/issue-start` 直前で実行される）。着手予定のない deferred Issue に対し着手レディネスを問うことは、**ゲートの目的と Issue の性質が乖離** している。

「完了条件をレビュー時点で機械的に検証可能にする」ために本文構造を改変するのは、deferred 性を損なう過剰設計であり、user 指示「無駄に複雑化し過ぎ。そもそも今 ready にする必要がない」に整合しない。

#### 3. 運用方針

- 本 Issue は計画記録ドキュメントとして open 維持
- trigger（GitLab 採用確定）発火時に **本 Issue を起点として着手 Issue を再構成** する。再構成された着手 Issue は通常通り `/issue-review-ready` を通す
- 本 Issue 自体は trigger 発火時の判断に委ねてクローズ or 継続 open

### EPIC 本文への反映

EPIC `local-pc5090-4` 本文に「`local-pc5090-11` / `local-pc5090-12`（deferred）は review-ready ゲート適用対象外」を明記し、後続レビューで同種の指摘が再発することを防ぐ。
