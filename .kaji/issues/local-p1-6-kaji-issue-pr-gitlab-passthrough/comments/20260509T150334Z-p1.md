### 設計の要点

1. **What**:
   - `normalize_id` に `gl:N` 形式と `provider_name="gitlab"` を追加（`ResolvedKind` に `"gitlab"` Literal を追加）
   - `kaji issue` dispatcher の GitLab 分岐（`glab issue` 転送、`comment` → `note` 名変換）
   - `kaji pr` Tier B 実装（`create`/`view`/`list`/`merge`/`comment`/`review`）
   - `kaji pr` Tier A 実装（`review-comments`/`reviews`/`reply-to-comment`）
   - 未対応 sub の明示エラー化（silent passthrough 禁止）
2. **Why**:
   - skill 側に GitHub/GitLab 分岐を持ち込まない（OQ-2 設計原則）
   - body は捨てない / note → approve/revoke 順序保証 / 未 approve 時 revoke skip
   - `kaji pr merge --squash`/`--rebase` を kaji 側で reject（CLAUDE.md `--no-ff only` 遵守）
3. **Constraints**:
   - 子 Issue #1（`GitLabProvider` 本体）完了が前提
   - GitHub mode は bit-exact に維持
   - `reply-to-comment` の `comment_id` は GitLab `discussion_id`+`note_id` を復元できる provider-local ID として kaji が独自発行

### テスト戦略

- **Medium**: `subprocess.run` モックで `glab issue/mr` 起動引数と順序（特に note → approve/revoke のシーケンス、未 approve 時 revoke skip、note 失敗時 approve スキップ）を検証
