## 概要

一次情報の記載と変更タイプ/テスト戦略の骨格はありますが、現状の設計では **GitLab provider で既存 skill 群をそのまま動かす contract** が成立していません。特に `reviews` 契約と gh 互換フラグ受理範囲、skill 側前提の扱いに未解消の穴があるため、実装着手可とは判断できません。

## 指摘事項 (Must Fix)

- [ ] **skill 不変前提が現状の skill 定義と衝突している**
  - 設計書は `skill ファイル (.claude/skills/) → 修正不要` としています（設計書 L158-L163, L296-L298）。しかし現行 skill は `gitlab` を受理していません。
  - `i-pr` は `provider_type` を `github` / `local` と明記し、`github` 以外を ABORT します（`.claude/skills/i-pr/SKILL.md` L37, L73-L118）。加えて `kaji pr create --base ...`、GitHub PR URL 正規表現、`gh auth status` / `kaji pr list --head <branch>` を前提にしています（同 L163-L235）。
  - `pr-fix` / `pr-verify` も `kaji pr list --search`, `--head`, `view --comments`, `reviews`, `review-comments` を前提にしています（`.claude/skills/pr-fix/SKILL.md` L120-L141, `.claude/skills/pr-verify/SKILL.md` L134-L155）。`issue-close` も provider 分岐が `github` / `local` の 2 分岐だけです（`.claude/skills/issue-close/SKILL.md` L62-L69）。
  - このままだと CLI を GitLab 対応しても workflow は end-to-end で動きません。skill を本 Issue で直すのか、別 Issue 依存に落として完了条件/影響範囲を書き換えるのかを設計上で明示してください。

- [ ] **gh 互換フラグ/サブコマンド対応表が不足しており、既存 skill 呼び出しを満たせていない**
  - `kaji issue` は `edit` を `glab issue <sub>` にそのまま流す設計ですが、GitLab CLI のサブコマンドは `update` です。`comment -> note` 変換だけでなく `edit -> update` 変換も必要です（設計書 L53-L57, L198-L208 / GitLab docs: `glab issue update`).
  - `kaji pr create` は skill から `--base` で呼ばれますが、設計書は `--title` / `--body` しか具体化していません。GitLab 側は `--target-branch` です（設計書 L64-L73, L226-L232 / `.claude/skills/i-pr/SKILL.md` L163-L184 / GitLab docs: `glab mr create -b, --target-branch`).
  - `kaji pr list` は skill から `--search` と `--head` で呼ばれます。`--search` は GitLab にありますが、`--head` はそのままではなく `--source-branch` 変換が必要です（`.claude/skills/pr-fix/SKILL.md` L120-L129, `.claude/skills/pr-verify/SKILL.md` L134-L143 / GitLab docs: `glab mr list --source-branch`).
  - `kaji pr view` は skill から `--comments` で呼ばれますが、設計書は `--json` / `--jq` 経路しか書いていません（設計書 L67, L91-L92, L228-L229 / `.claude/skills/pr-fix/SKILL.md` L139, `.claude/skills/pr-verify/SKILL.md` L153 / GitLab docs: `glab mr view --comments`).
  - `skill 側 contract は GitHub 互換 subset を正本` を守るなら、「どの gh 互換引数を kaji が受理し、どの glab 引数へ写像するか」を表で固定してください。現状の記述では実装者依存の余地が大きすぎます。

- [ ] **`kaji pr reviews` の contract が approvals API だけでは成立しない**
  - 設計書は `reviews` を `GET .../approvals`（+ approval_state）から GitHub `pulls/<N>/reviews` 互換 subset に変換するとしています（設計書 L77-L80, L231-L236, L309-L311）。
  - しかし approvals API で取れるのは approval 情報であり、`pr-fix` / `pr-verify` が参照している `.state` と `.body` をそのまま再構成できません（`.claude/skills/pr-fix/SKILL.md` L139-L141, `.claude/skills/pr-verify/SKILL.md` L153-L155）。特に `review --request-changes` は note + revoke で実装する設計ですが、revoke 後に approvals API から `CHANGES_REQUESTED` 相当の review state は得られません。
  - つまり現設計のままでは「GitHub 互換 subset」の定義が曖昧で、skill が期待する review 一覧として使えません。`reviews` を approval 履歴ではなく「top-level review note + approval/unapproval 操作の統合ビュー」として再定義するのか、skill 側を `review-comments` / top-level comments 中心に変えるのか、contract を先に固定してください。
