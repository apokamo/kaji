## 対応済み

### Must Fix 1: GitLab PR read 経路を実 glab CLI 契約に整合（A: 対応）

**指摘の通り、実 glab CLI には該当 flag が存在しないことを確認。**

\`\`\`
$ glab mr view --help  → --output flag は存在しない（--comments / --page / --per-page / --system-logs / --web のみ）
$ glab mr list --help  → -F flag / --output flag ともに存在しない
\`\`\`

- **`kaji_harness/providers/gitlab.py`**: PR read 用の provider helper を追加し、既存 `view_issue` / `list_issues` / `list_labels` の `glab api` 経由パターンに揃えた:
  - `get_mr_view_payload(mr_iid)` → `glab api projects/<encoded>/merge_requests/<iid>`
  - `list_mrs_payload(state=, source_branch=, target_branch=, search=, per_page=)` → `glab api projects/<encoded>/merge_requests?...` （URL encode + per_page クランプ）
- **`kaji_harness/cli_main.py`**: `_gitlab_pr_view` / `_gitlab_pr_list` を上記 helper 経由に書き換え。`provider._run_glab(\"mr\", \"view\", ..., \"--output\", \"json\")` および `glab mr list -F json` 直叩き経路を撤去。
- 結果として skill `pr-fix` / `pr-verify` の `kaji pr list --search`、`--head`、`pr view --json` 経路は実 glab で動作する `glab api` ルートで成立する。

### Must Fix 2: create 系に `--yes` 注入（A: 対応）

**`glab issue create --help` / `glab mr create --help` 双方に `-y, --yes` 存在を確認**。skill (issue-create / i-pr) は非対話実行前提なので prompt 待ちで stuck するリスクは指摘通り。

- **`_gitlab_pr_create`**: `glab_args.append(\"--yes\")` を末尾追加（無条件注入）
- **`_handle_issue_gitlab`**: sub == \"create\" のときのみ `--yes` を注入。user が既に `--yes` / `-y` を指定していれば二重注入しない（`assert cmd.count(\"--yes\") == 1` でテスト）
