# レビュー指摘への対応報告

レビューありがとうございます。Must Fix 2 件をいずれも受容して修正しました。

## 対応済み (2 件)

### MF-1: MAX_PAGES 境界の false positive abort

- 修正内容: `kaji_harness/sync.py:130-170` の `_fetch_open_issues_paginated` を再構成。`while True` ループ内で `payload` を取得し、空配列なら break、`page > _MAX_PAGES` でデータが来た場合のみ `SyncError` を投げる。これにより「ちょうど `MAX_PAGES × per_page` 件 (= 20,000)」は **MAX_PAGES + 1 page 目が空配列であることを終端確認** したうえで成功扱いになる。
- 回帰テスト: `tests/test_sync_from_gitlab.py::TestPagination::test_exactly_max_pages_succeeds`（`MAX_PAGES=3` で 100×3=300 件 round-trip 検証）。既存の abort テストも 3 page 目もフルなら raise の semantics に揃えた。

### MF-2: from-gitlab の出力契約と設計書の乖離

- 修正内容: `kaji_harness/sync.py:299-324` の phase 3 で各 entry を処理しながら `(newly_added, updated, unchanged_signature)` を計上。新規 helper `_read_existing_issue_payload()` で既存 wrapper の `issue` field を読み、新 payload と dict 比較する。出力は設計書 § インターフェース 1 の通り `Wrote N issues to .kaji/cache/ (X newly added, Y updated, Z unchanged signature).`。
- 回帰テスト: `tests/test_sync_from_gitlab.py::TestCliSyncFromGitLab::test_wrote_breakdown_matches_design`（初回 sync = 全件 newly added、再 sync で 1 件のみ title 変更 = 1 updated + 2 unchanged signature を検証）。
