### 設計の要点

1. **What**: `provider.type='local'` 配下から GitLab Issue を `gl:N` で参照できるよう、`kaji sync from-gitlab` で GitLab project の open Issue 全件を `.kaji/cache/gl-<iid>.json` に atomic write する。`kaji sync status` で cache 状態を表示し、`kaji issue list` を local + cached 統合表示する。
2. **Why**: EPIC `local-pc5090-4` 子 Issue #4。`local-pc5090-5/6` で GitLabProvider 経路は確立済だが、local 配下から `gl:42` を読む cache populate 経路が未整備。
3. **Constraints**:
   - cache layout: `.kaji/cache/gl-<iid>.json`（既存 `gh:` 用 `.kaji/cache/issues/<n>.json` とは分離。改名なし）
   - pagination: `?per_page=100` + `?page=N` のオフセット pagination ループ。空配列 / `<100` 件で終了。MAX_PAGES=200 サニティリミット
   - 未実装 flag (`--include-closed` / `--state` / `--since`) は **silently ignore せず exit 2 で fail-fast**（完了条件）
   - closed 化された entry は cache に残す（fetch 結果に含まれない gl-*.json を delete しない）
   - sync_meta は **全 page fetch 完了後にのみ書く**（途中失敗時は古い meta を保持 ＝ 最終 sync 時刻が嘘にならない）

### 主要モジュール

- `kaji_harness/sync.py`（新設） — `sync_from_gitlab()` / `read_sync_status()` / `SyncResult` / `SyncStatus` / `SyncError`
- `kaji_harness/cli_main.py` — `_register_sync()` / `cmd_sync_from_gitlab()` / `cmd_sync_status()`
- `kaji_harness/providers/local.py` — `view_cached_gitlab_issue()` / `list_issues()` の cache 統合 / `_gitlab_cache_path` helper
- `kaji issue view` dispatcher — `rid.raw.startswith("gl:")` で gh/gl 分岐
