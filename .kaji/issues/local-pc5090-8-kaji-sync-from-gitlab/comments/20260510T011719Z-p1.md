## 概要

設計書は feature 設計として、CLI 契約・使用例・一次情報・影響ドキュメント・S/M/L テスト観点まで揃っており、全体像は明確です。一方で、**失敗時の同期挙動**と**closed 化した cache entry の見せ方**に内部矛盾があり、このままでは実装者がどちらの契約を守るべきか決められません。実装前に設計上の契約を 1 つに揃える必要があります。

## 指摘事項 (Must Fix)

- [ ] **partial sync の契約がデータフローと矛盾している**
  - 設計本文では `sync_from_gitlab()` が `fetch_open_issues_paginated()` で全ページ取得した後に `_write_cache_files()` を呼ぶ流れです（`draft/design/issue-local-pc5090-8-kaji-sync-from-gitlab.md:591-597`）。
  - しかし失敗時仕様では「1 ページ目成功 + 2 ページ目失敗なら 1 ページ目分の cache は残る、`.sync-meta.json` は書かない」としています（同:501-505, 846-847）。この挙動は、現在のデータフローだと `_fetch_open_issues_paginated()` が例外で戻らないため実現できません。
  - **必要な修正**: 「ページごとに fetch→atomic write し、最後に meta だけ commit する」方式に設計を寄せるか、逆に「全 fetch 成功までは 1 件も書かない」方式に統一するかを明記してください。テスト戦略も同じ契約に合わせて修正が必要です。
  - **一次情報との関連**: GitLab REST は offset pagination を許容していますが、partial failure 時のローカル整合性は一次情報では決まらないため、設計書側で不変条件を一意に定義する必要があります。

- [ ] **closed 化した GitLab issue を cache に残す契約と `kaji issue list` の状態表示が両立していない**
  - 設計では「open 全件のみ取得し、取得結果に含まれなくなった `gl-*.json` は残すが更新しない」としています（同:602-604, 701-710）。
  - 一方で `list_issues()` 側は cache JSON 内の `issue.state` だけを見て `open/closed` を判定し、`--state closed` の Medium テストまで要求しています（同:729-761, 855-859）。
  - これだと upstream で closed になった issue は cache 上では最後に取得した `state="opened"` のまま残るため、**既定の `kaji issue list` で閉じた issue を open と誤表示**し続けます。さらに `--state closed` では拾えません。
  - **必要な修正**: stale entry をどう扱うかを設計で明示してください。例えば `fetched_at` と `.sync-meta.json:last_sync_at` の比較で「今回の open sync に現れなかった entry」を `stale/closed相当` とみなして open 表示から除外する、あるいは cache 保持先を active/archive に分ける、などのルールが必要です。
  - **一次情報との関連**: GitLab Issues API は `state=opened|closed|all` を返せますが、本設計は `opened` のみ取得です。したがって closed 判定はローカル設計で補う必要があります。

## 改善提案 (Should Fix)

- **`MAX_PAGES=200` の扱いを契約として明示する**
  - 設計は「open Issue 全件取得」を完了条件に置きつつ、内部では `MAX_PAGES=200` を設けています（同:612-634）。現状のままだと 20,000 件超の project で仕様違反になります。
  - 運用上十分と判断するなら、Issue 本文/設計書の完了条件にその上限を明記してください。そうでないなら hard cap は置かず、GitLab API の pagination 契約に従って取り切る方が筋が通ります。
