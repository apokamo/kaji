# 設計修正報告（cycle 1/3）

## 対応済み (2 件)

### MF-1: partial sync 矛盾の解消（all-or-nothing 化）

- 設計書 § sync の主要データフロー を **3 phase** に書き直し:
  - phase 1 fetch: 全 page 完了まで in-memory。**任意 page の失敗 → cache を一切触らずに return**
  - phase 2 stale 判定: fetched_iids vs existing_iids の集合差分を計算（IO なし）
  - phase 3 write: fresh entry overwrite → stale entry の wrapper 更新 → `.sync-meta.json`
- § エラーケース :501 の partial-sync 記述を削除し、**fetch phase 失敗時は cache に何も書かない** と書き換え
- § Medium テスト :846-847 の「1 ページ目分の cache は残る」を **「1 ページ目分の cache も書かれない」** に修正（all-or-nothing 検証として明記）
- 関連: 1 page 目成功 + 2 page 目失敗時に既存 cache が `mtime` 含めて 1 byte も触られないことを Medium テストで検証する追加項目を追記

### MF-2: stale entry の開閉表示矛盾の解消（kaji_local.is_stale フラグ）

- cache wrapper schema に `kaji_local: { is_stale, last_seen_at, staled_at }` を追加（schema_version=1 の一部として最初から導入）
- **issue.state は触らない** invariant を明示（GitLab 由来 field の正本性を維持）
- sync 時の挙動を § stale 化された entry の扱い に明記:
  - fetch 結果に含まれない既存 entry → `_mark_cache_stale()` で `is_stale=true / staled_at=now` に遷移、issue 本体は不変
  - 既に `is_stale=true` の entry は再 write しない（`staled_at` / `last_seen_at` の上書き防止）
  - 再 open された issue が次回 fetch に含まれた場合、`_write_fresh_cache_file()` が wrapper を完全置換して fresh に復帰
- `list_issues()` の表示 state 決定ルールを表形式で明文化:
  - `is_stale=true` → **無条件 closed 扱い**（GitLab `issue.state` に依らない）
  - `is_stale=false` AND `issue.state="opened"` → open
  - それ以外 → closed
- 後方互換: `kaji_local` field を持たない旧 schema cache は `is_stale=false` 扱い（user が手動投入した既存 cache を壊さない）
- Medium テストに以下を追加:
  - `kaji issue list`（既定 = `--state open`）で stale 化された entry が **出ない** こと
  - `kaji issue list --state closed` で local closed + stale entry が出ること
  - `kaji issue list --state all` で全件出ること
  - sync round-trip で `is_stale=false → true` 遷移、復帰時の `true → false` 遷移
  - 既に stale な entry の `staled_at` / `last_seen_at` が再 sync で書き換わらない invariant
