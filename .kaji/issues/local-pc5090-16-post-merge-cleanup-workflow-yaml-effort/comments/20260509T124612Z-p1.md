## 指摘事項 (Must Fix)

- [ ] **`_commit_local_issue_change` は現設計だと atomic にならない**
  - 設計書では `git add <paths> -> git commit -m ...` とした上で、「既に staged だった他ファイルはそのまま残し、ABORT しない」としています（設計書 lines 190-193）。しかし `git commit -m ...` は index 上の staged change をまとめて commit するため、whitelist 対象外の staged file も同じ commit に混入します。
  - 実検証でも、temp repo で target file と `other.txt` を staged した状態で `git commit -m test` を実行すると、HEAD には両方が入りました。設計書 lines 345-346 の「latest commit に `.kaji/issues/<id>/...` のみ含まれる」という atomicity 期待は、この実装方針では成立しません。
  - **必要な修正**: `--commit` 実行前に index が空であることを検査して ABORT する、または path 限定 commit を本当に保証できる方式（別 index 利用、対象 path 以外を commit 対象に含めない明示手順）へ設計を修正してください。

- [ ] **`issue-close` の救済 whitelist が記述より広く、`comments/*.md` 以外も拾う**
  - 設計書は「救済対象を `issue.md` と `comments/*.md` に限定する」と説明しています（lines 217-226）が、実際の疑似コードは `git add ".kaji/issues/[issue_id]-*/comments/"` で comments ディレクトリ全体を stage します（line 218）。
  - 実検証でも、temp repo で `.kaji/issues/.../comments/note.txt` を置いた状態でその `git add` を行うと、`note.txt` も stage されました。これでは「local issue 永続化由来のみ」を path whitelist で機械的に近似する、という設計判断が崩れます。
  - **必要な修正**: rescue でも対象 pathspec を実際に `issue.md` と generated comment markdown のみに絞ってください。少なくとも `comments/` ディレクトリ丸ごと add は不可です。
