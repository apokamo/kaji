### 設計の要点

1. **What**:
   - **A**: `kaji_harness/workflow.py::_parse_workflow` に agent 別 allowed values を辞書で保持する runtime validator を追加し、許容外 effort を `WorkflowValidationError` で reject。`feature-development-local.yaml` の line 27/46/67/89 の `xHigh`/`High` を `xhigh`/`high` に修正
   - **B**: `kaji issue {comment,edit}` に `--commit` フラグを追加し永続化と commit を atomic 化。`issue-close` skill の Step 2 を「path 範囲 whitelist 制約 + 限定 glob add + 残差再検証」の 3 段ガードに書き換え

2. **Why**:
   - A: `models.py` の dataclass `Literal` hint は静的解析専用で runtime 検証にならない（[Python typing 仕様](https://docs.python.org/3/library/typing.html#typing.Literal) より）。`workflow.py` parse 時の検証で機械的に reject する必要がある
   - B: 永続化と commit が分離している現状では skill 群の責務として commit を強制できず、`issue-close` の ABORT ガードを Claude が逸脱するインセンティブが生じる。CLI 側で atomic commit を提供することで skill 側責務を確実化する

3. **Constraints**:
   - 後方互換: A の validator は既存全 workflow YAML（小文字のみ）で PASS。B の `--commit` 不指定時の挙動は変更前と bit-exact 一致
   - allowed values 辞書未登録の agent（gemini 等）は validation skip（passthrough）で拡張性を維持

### 設計判断（Issue 提示の選択肢への回答）

- **A の許容値方針**: **(A2) agent 別 allowed values** を採択。(A1) の共通 subset only は claude `max` / codex `none/minimal` を将来も使えなくする
- **B の commit 動線対象範囲**: **(B2) `comment + edit`** を採択。(B1) は `edit` 由来の `issue.md` 蓄積を取りこぼす。(B3) の `close` は `issue-close` skill 内で commit が完結する既存設計を維持
- **救済機構**: 完全廃止せず、3 段ガード（path 範囲制約 / whitelist glob add / 残差再検証）で残す。条件 2 「永続化由来のみ」は **`issue.md` と `comments/*.md` の glob 限定で機械的に近似**（git diff の行内容では判定不可能なため、追加対象パスを限定する設計）
