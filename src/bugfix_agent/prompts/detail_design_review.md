# DETAIL_DESIGN Review Prompt

Issue ${issue_url} の `## Bugfix agent DETAIL_DESIGN` セクションをレビューしてください。

## タスク

1. `gh issue view` で最新の Issue 本文を取得
2. DETAIL_DESIGN セクションの必須アウトプットを確認
3. IMPLEMENT ステートに移行可能か判定

## 完了条件チェックリスト

| # | 項目 | 確認内容 |
|---|------|----------|
| 1 | **変更計画** | 対象ファイル・関数と計画する変更内容が詳細に記載されている |
| 2 | **実装手順** | ステップバイステップの実装手順が記載されている |
| 3 | **テストケース一覧** | テストケースが列挙されている（目的、入力、期待結果） |
| 4 | **補足情報** | 追加の実装注意事項がある（なくても可） |

## 禁止事項

**次ステート以降の責務を新規に実行しない**
- 例：実装、テスト追加、PR作成

**既に完了した責務を再実行しない**
- 例：調査を再実行、再現テストを再作成

※ 記載内容の検証（ソースコード確認、整合性チェック）は**許可**されています。

## 出力形式

```markdown
### DETAIL_DESIGN Review Result

#### 検証内容
- <実施した検証と結果を具体的に記載>

#### チェックリスト
- 変更計画(対象/手順): <OK/NG + 具体的根拠>
- 実装手順: <OK/NG + 具体的根拠>
- テストケース一覧: <OK/NG + 具体的根拠>
- 補足情報: <OK/NG or N/A>

## VERDICT
- Result: PASS | RETRY
- Reason: <判定理由>
- Evidence: <具体的な判断根拠>
- Suggestion: <RETRY時: 具体的な設計修正指示>
```

## 判定ガイドライン

| 状況 | VERDICT | 次のステート |
|------|---------|-------------|
| 実装に必要な情報がすべて揃い、矛盾がない | PASS | IMPLEMENT |
| 不足・矛盾があり設計の見直しが必要 | RETRY | DETAIL_DESIGN |

## レポート方法

1. `gh issue comment` で Issue にコメント投稿（VERDICT判定）
2. **PASS判定時のみ**: 共通ルールに従い `gh issue edit` で Issue 本文を更新
