# [設計] ドキュメント同期ステップの追加

Issue: #12

## 概要

Issue駆動開発ワークフローに「ドキュメント同期」フェーズを追加し、実装完了後・PR作成前に設計書と関連ドキュメントの整合性を確認・更新する。

## 背景・目的

### 現状の課題

1. 設計レビュー・修正サイクルで `draft/design/` が更新されても、関連ドキュメント（ADR、architecture.md等）に反映されない
2. PR作成（ルーチン作業）とドキュメント同期（判断が必要）が混在すると、整合性確認が漏れやすい

### 解決策

コードレビュー完了後、PR作成前に「ドキュメント同期」フェーズを挿入:

```
設計 → 設計レビュー → 実装 → コードレビュー → ドキュメント同期 → PR作成
```

## インターフェース

### 入力

- `issue-number`: Issue番号（必須）

### 出力

- 更新された関連ドキュメント（ADR、architecture.md等）
- コミット

### 使用例

```bash
/issue-sync-docs 12
```

## 制約・前提条件

- `/issue-start` が実行済みであること
- コードレビューが完了していること（推奨）
- 自動差分検出はスコープ外（手動確認で十分）

## 方針

### 1. DEVELOPMENT_WORKFLOW.md の更新

フェーズ番号を繰り上げて「5. ドキュメント同期」として独立:

```
フェーズ概要テーブル:
| 5. ドキュメント同期 | `/issue-sync-docs` | ADR・architecture.md 等の更新 |
| 6. PR作成 | `/issue-pr` | PR（設計書をPR本文に転記後、draft/削除） |
| 7. 完了 | `/issue-close` | PRマージ + worktree削除 |
```

### 2. `/issue-sync-docs` スキルの作成

#### 実行手順

1. Worktree情報取得・移動
2. 設計書(draft/design/)の確認
3. **判断チェックリストの提示**（後述）
4. 必要に応じて更新
5. コミット
6. Issueにコメント

#### 判断チェックリスト

スキル実行時に以下のチェックリストを提示し、該当項目があれば更新を促す:

```markdown
## ドキュメント更新チェックリスト

設計書の内容を確認し、以下に該当する場合は対応するドキュメントを更新してください。

### ADR (Architecture Decision Record)
- [ ] 新しい技術選定・ライブラリ採用の決定があるか？
- [ ] 既存のアーキテクチャパターンを変更する決定があるか？
- [ ] 重要なトレードオフを伴う設計判断があるか？

### architecture.md
- [ ] システム構成・コンポーネント構造に変更があるか？
- [ ] 新しいモジュール・レイヤーを追加したか？
- [ ] 依存関係に変更があるか？

### guides/
- [ ] 開発者向けの新しい手順・ワークフローがあるか？
- [ ] 既存の運用手順に変更があるか？
- [ ] 新しいツール・コマンドの使用方法があるか？
```

#### スキップ条件

以下の場合、ドキュメント更新は不要（チェックリストで該当なしの場合）:

- **バグ修正**: 既存の動作を修正するだけで設計変更なし
- **軽微なリファクタ**: 内部実装の改善で外部仕様・構造に影響なし
- **テスト追加**: テストコードのみの変更
- **ドキュメント修正**: typo修正等の軽微な変更

スキップ時も「確認済み・更新不要」としてIssueにコメントを残す。

### 3. スキル説明の更新

各スキルの「いつ使うか」セクションにワークフロー内の位置を明記。

## 検証観点

- 正常系: `/issue-sync-docs` 実行後、チェックリストに基づき関連ドキュメントが更新されコミットされる
- 境界値: チェックリストで該当なしの場合、「確認済み・更新不要」としてコメントのみ投稿
- 整合性: DEVELOPMENT_WORKFLOW.md のフローとスキル一覧が一致する
