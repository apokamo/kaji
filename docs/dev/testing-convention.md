# Testing Convention

テスト規約。設計・実装・レビュー時に参照する。

## テストサイズ定義

テストはリソース制約（外部依存の有無）によって S/M/L の3サイズに分類する。

| サイズ | 名称 | 特徴 | 実行速度 |
|--------|------|------|---------|
| **S** | Small | 外部依存なし。純粋なロジック・バリデーション・マッピング | 高速（ms） |
| **M** | Medium | ファイルI/O・DB・内部サービスとの結合 | 中速（秒） |
| **L** | Large | 実API・E2E・外部サービス疎通 | 低速（秒〜分） |

### マーカー

```python
@pytest.mark.small
@pytest.mark.medium
@pytest.mark.large
```

### 判定基準

```
外部 API / 実サービス疎通あり → Large
DB / ファイル / 内部サービス結合あり → Medium
それ以外（純粋関数・モック完結） → Small
```

## S/M/L 全サイズ実装の原則

> **設計段階で全サイズを定義し、省略の余地を排除する。**

設計書の「テスト戦略」には Small / Medium / Large それぞれの検証対象を必ず記載すること。
実装時も同様に全サイズのテストを作成する。

## スキップ判定基準

スキップが許容されるのは **物理的に作成不可** な場合のみ。

| 許容されるスキップ理由 | 例 |
|----------------------|-----|
| スキーマ未整備で Medium が書けない | DBテーブルが存在しない |
| API アクセス手段が存在しない | 外部サービスのサンドボックスが提供されていない |

### 不正当な省略理由（自動的に Changes Requested）

以下は「物理的不可能」に該当しないため、スキップは認められない:

| 不正当な理由 | 対応 |
|------------|------|
| 「実行時間が長い」 | サンプル期間等で短縮可能 |
| 「API キーがない」「DB が起動していない」 | 環境不備は修正対象。ユーザーに相談するか自分で解決 |
| 「Small テストで十分カバーされている」 | テストサイズごとに検証対象が異なる |
| 「軽微な変更なのでテスト不要」 | 変更の大小はスキップ理由にならない |
| 「Large テストはステージング環境で検証」 | CI で実行できる構成にすること |

## AI のテスト省略傾向への警告

> **AI には M/L テストを省略する強い傾向がある。この傾向を自覚すること。**
>
> 「このテストは不要では？」「Small で十分では？」と思った場合、**それはほぼ確実に誤りである。**

### 禁止事項

- ❌ Large テストを「ステージング環境での検証」として後回しにする
- ❌ 「実行時間が長い」を理由に M/L テストを省略する
- ❌ 「API キーがない」などの環境不備を理由に M/L テストをスキップする
- ❌ 「Small テストで十分カバーされている」と判断して M/L を省略する
- ❌ 「軽微な変更」を理由にテストサイズを落とす

## 設計書テスト戦略セクションの書き方

```markdown
## テスト戦略

> **CRITICAL**: S/M/L すべてのサイズのテスト方針を定義すること。

### Small テスト
- (検証対象を列挙: 単体ロジック、バリデーション、マッピング等)

### Medium テスト
- (検証対象を列挙: DB連携、内部サービス結合等)

### Large テスト
- (検証対象を列挙: 実API疎通、E2Eデータフロー等)

### スキップするサイズ（該当する場合のみ）
- サイズ: (物理的に作成不可な理由を明記。「実行時間」「環境依存」は不正当)
```

## レビュー時のチェックリスト

```
- [ ] Small テストの検証対象が定義されているか
- [ ] Medium テストの検証対象が定義されているか
- [ ] Large テストの検証対象が定義されているか
- [ ] スキップするサイズがある場合、理由は物理的不可能に該当するか
```
