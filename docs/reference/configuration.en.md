# Configuration Reference (`.kaji/config.toml`)

> 日本語版: [configuration.md](configuration.md)

The authoritative **Source of Truth** for every section / key of kaji's config file
`.kaji/config.toml` and its overlay `.kaji/config.local.toml`. Each key's type / default /
validation rule / behavior is taken directly from `kaji_harness/config.py` and
`kaji_harness/local_init.py`.

For minimal, in-context setup examples, see the CLI guides
([GitHub Mode](../cli-guides/github-mode.md) / [Local Mode](../cli-guides/local-mode.md) /
[Interactive Terminal Runner](../cli-guides/interactive-terminal-runner.md)). This document is
scoped to the exhaustive key specification only.

## Overview

- `.kaji/config.toml` also acts as the **marker that identifies the repository root**. kaji walks
  upward from `--workdir` (or the cwd) looking for `.kaji/config.toml`, and treats the directory
  containing `.kaji/` as the repository root.
- This file is the Source of Truth for the config spec. When updating a key's type / default /
  validation, update this document first and keep the how-to / CLI guides limited to minimal
  examples and links.

## File roles

| File | git | Role |
|------|-----|------|
| `.kaji/config.toml` | tracked | Repository default. Holds the repository-standard settings |
| `.kaji/config.local.toml` | gitignored | Per-user overlay. Overrides tracked values per key in `[execution]` / `[provider]` only (see [Overlay merge rule](#overlay-merge-rule)) |

- The overlay is generated by `kaji local init` (see [Local Mode CLI Guide](../cli-guides/local-mode.md) § 2).
- `git worktree add` only **checks out tracked files**, so the gitignored overlay is **not carried
  over to new worktrees**. See
  [Git Worktree guide](../guides/git-worktree.md#provider-overlay-kajiconfiglocaltoml-は新規-worktree-に引き継がれない)
  for details and the workaround.

## Discovery rule

`KajiConfig.discover()` (`kaji_harness/config.py`) resolves the repository root as follows:

1. Start from `--workdir` (or the cwd if unspecified).
2. Walk upward and take the first directory that contains `.kaji/config.toml` as the repository root.
3. If the filesystem root is reached without a hit, stop with `ConfigNotFoundError`.

`kaji run`'s config discovery start directory can be set with `--workdir` (distinct from the
workflow YAML `workdir`; see [Workflow authoring](../dev/workflow-authoring.md) § run commands).

## Overlay merge rule

The overlay (`.kaji/config.local.toml`) can override tracked values in **only the `[execution]` and
`[provider]` sections**, both merged **per key within the top-level section**.

- `[execution]`: a key in the overlay overrides the same key in tracked. Keys not present in the
  overlay keep their tracked values.
- `[provider]`: `type` / `[provider.github]` / `[provider.local]` can be **deep-1 merged**. If the
  overlay sets `type = "local"`, the provider switches even when tracked is `type = "github"`, and
  the `[provider.github]` / `[provider.local]` subtables are merged per key.
- Only when neither tracked nor overlay defines `[provider]` does the loader return the provider as `None`.
- `[paths]` (`artifacts_dir` / `skill_dir` / `worktree_prefix`) is **not** overlaid. `PathsConfig` is
  built solely from tracked `.kaji/config.toml`; the overlay is passed only to `[execution]` /
  `[provider]` parsing (`config.py:141-154`). A `[paths]` table in the overlay is ignored.

Validation error messages point at the file that actually defined the key (tracked / overlay) only
for each `[execution]` key and for `provider.local.machine_id` (`config.py:213-249` `source()`,
`config.py:369-374` `source_path`). For the rest of `[provider]` (e.g. `type` / `[provider.github]`
type errors), the message uses `ConfigLoadError(path, ...)` and therefore points at the tracked
`.kaji/config.toml` even when the value came from the overlay (`config.py:323-354`).

## Section / key specification

A key's "Default" column is the value the config loader adopts at parse time. For keys where the
loader default differs from the **effective fallback** applied by the provider layer at path
computation (`worktree_prefix`), both are described separately.

### `[paths]`

| key | Required/Optional | Type | Default | Validation | Source |
|-----|-------------------|------|---------|------------|--------|
| `artifacts_dir` | Required | str | — (unset is an error) | Relative paths must not contain `..` (prevents repo-root escape). Absolute paths / `~` expansion are allowed | `config.py:118-125`, `397-417` |
| `skill_dir` | Required | str | — (unset is an error) | Relative only. No absolute paths, no `..` | `config.py:126-133`, `419-434` |
| `worktree_prefix` | Optional | str | `""` (unset) | When non-empty, a single safe segment (`[A-Za-z0-9._-]+`; no separators / whitespace / `..` / absolute) | `config.py:134-140`, `436-452` |

> **`worktree_prefix`: "config default" vs "effective fallback"**:
>
> - **Config default (loader layer)**: when unset, `config.paths.worktree_prefix` is the **empty
>   string `""`** (`config.py:134`, `paths_data.get("worktree_prefix", "")`).
> - **Effective fallback (provider layer)**: the worktree dir name is computed with
>   `worktree_prefix or "kaji"` (`kaji_harness/providers/context.py:94` /
>   `kaji_harness/worktree_discovery.py:92`), so the empty string falls back to `"kaji"` at path computation.
> - As a result, the worktree dir name is `kaji-<branch_prefix>-<id>`. Whether you set
>   `worktree_prefix = "kaji"` explicitly in `.kaji/config.toml` or leave it unset, the **generated
>   worktree path is identical**. Only the field value (`config.paths.worktree_prefix`) changes from
>   `"" → "kaji"`.

A relative `artifacts_dir` is resolved against the main worktree (the worktree that has
`provider.<type>.default_branch` checked out) — Issue #177, see
[Workflow authoring](../dev/workflow-authoring.md) § prerequisites.

### `[execution]`

| key | Required/Optional | Type | Default | Validation | Source |
|-----|-------------------|------|---------|------------|--------|
| `default_timeout` | Required | int | — (unset is an error) | Integer `> 0` (bool not allowed) | `config.py:219-231` |
| `agent_runner` | Optional | `"headless"` \| `"interactive_terminal"` | `"headless"` | Out-of-enum is `ConfigLoadError` | `config.py:233-244` |
| `interactive_terminal_close_on_verdict` | Optional | bool | `true` | Non-bool is `ConfigLoadError` | `config.py:246-252` |

- `agent_runner` selects whether agent steps launch via a headless CLI or an interactive CLI inside a
  tmux pane. For `interactive_terminal` behavior, CLI options, and precedence, see the
  [Interactive Terminal Runner guide](../cli-guides/interactive-terminal-runner.md).
- `interactive_terminal_close_on_verdict` takes effect only when
  `agent_runner = "interactive_terminal"` (whether to close the pane after a verdict is detected). It
  is inert under headless operation.

The `timeout` resolution order is step.timeout → workflow.default_timeout →
`config.execution.default_timeout` (see [Workflow authoring](../dev/workflow-authoring.md) § step fields).

### `[provider]`

The `[provider]` section is optional at the config loader layer: when neither tracked nor overlay
defines it, the loader returns the provider as `None` (`config.py:303-304`). However, the provider
resolution path of `kaji issue` / `kaji pr` / `kaji run` requires `[provider]` (Phase 3-e; unset
exits 2). For the detailed migration, see [Local Mode CLI Guide](../cli-guides/local-mode.md) § 10.

| key | Required/Optional | Type | Default | Validation | Source |
|-----|-------------------|------|---------|------------|--------|
| `type` | Required if `[provider]` is present | `"github"` \| `"local"` | — (required when `[provider]` is written) | Out-of-enum is `ConfigLoadError` | `config.py:320-329` |

### `[provider.github]`

| key | Required/Optional | Type | Default | Validation | Source |
|-----|-------------------|------|---------|------------|--------|
| `repo` | Optional | str (`owner/name`) | `""` | Non-str is `ConfigLoadError`. Do not add an `https://` prefix or `.git` suffix | `config.py:334-336` |
| `default_branch` | Optional | str | `"main"` | Non-str is `ConfigLoadError` | `config.py:337-339` |
| `git_remote` | Optional | str | `"origin"` | Non-str is `ConfigLoadError` | `config.py:340-342` |

- `repo` is passed to `gh --repo <owner>/<name>` / `gh api repos/<owner>/<name>/...`. Setting it keeps
  the write target stable even when the worktree's git remote points at a fork.
- `git_remote` is the git remote name targeted by `git push` / `git fetch` inside skills. In release
  operation, `provider.github.git_remote` is used to verify the remote name pointing at GitHub (see
  [Release Runbook](../operations/release/runbook.md)).

### `[provider.local]`

| key | Required/Optional | Type | Default | Validation | Source |
|-----|-------------------|------|---------|------------|--------|
| `machine_id` | Optional | str | `""` | When non-empty, `[a-z0-9]{1,16}` (lowercase alphanumeric only, no hyphen, max 16 chars). Violations are `ConfigLoadError` | `config.py:352-377` |
| `default_branch` | Optional | str | `"main"` | Non-str is `ConfigLoadError` | `config.py:378-380` |
| `git_remote` | Optional | str | `"origin"` | Non-str is `ConfigLoadError` | `config.py:381-383` |

## local provider / overlay handling

The local provider is not the standard operation mode of this repository. For value specs see
`[provider.local]` above; for operational how-to see the
[Local Mode CLI Guide](../cli-guides/local-mode.md).

`kaji local init` writes the following 3 values to the overlay (`.kaji/config.local.toml`)
(`kaji_harness/local_init.py:243-258`):

- `[provider] type = "local"`
- `[provider.local] machine_id = "<resolved value>"`
- `[provider.local] default_branch = "<--default-branch | main>"`

`machine_id` resolution order (`local_init.py:215-240`):

1. Explicit `--machine-id <name>` (a `[a-z0-9]{1,16}` violation exits 2).
2. Sanitized `socket.gethostname()` (lowercase + alphanumeric + truncated to 16 chars), adopted if it
   does not collide with existing local Issues.
3. Fallback to `pc1` / `pc2` / … (the smallest value that does not collide with existing
   `.kaji/issues/local-*`).

## Example

A minimal configuration for GitHub-standard operation (`type = "github"` / headless). The
behavior-relevant primary values are made explicit instead of relying on implicit defaults. For each
key's detailed spec, see the corresponding section above.

```toml
# .kaji/config.toml (tracked, repository default)
# Source of Truth for the config spec: docs/reference/configuration.md

[paths]
artifacts_dir = ".kaji-artifacts"
skill_dir = ".claude/skills"
worktree_prefix = "kaji"            # leading segment of the worktree dir name (<prefix>-<branch_prefix>-<id>)

[execution]
default_timeout = 2400
agent_runner = "headless"           # "headless" (default) | "interactive_terminal"
# interactive_terminal_close_on_verdict = true   # only takes effect under interactive_terminal

[provider]
type = "github"

[provider.github]
repo = "apokamo/kaji"
default_branch = "main"
git_remote = "origin"
```
