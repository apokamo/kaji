# ADR 008: 後方互換レイヤを提供しない（BREAKING 明示ポリシー）

## ステータス

提案 — Issue #261 の設計レビュー（`/issue-review-design`）で承認を判定する。
承認後、本ステータス行を「承認（レビュー通過日）」に更新する。

## コンテキスト

- kaji は単一メンテナ + starter 経由の少数利用者で、pre-1.0。互換レイヤ
  （旧フォーマット読み取り・フォールバック・バージョン分岐）を書くコスト・保守するコスト・
  互換レイヤ自体のバグは、破壊的変更の一回性コストより確実に高い。
- 実例: `issue-design` の BACK 再入検出（Issue #259 項目 2 → #261）では、producer / consumer
  契約を SKILL.md の散文で二重管理した結果、不一致が silent に発生し検出が一度も機能しなかった。
  互換コードは同種の「死んだ読み取り器」を増やす温床になる。
- 「破壊的変更は in-flight workflow がないタイミングで入れる」という運用案は、kaji を導入した
  下流リポジトリの workflow 状態を観測できないため成立しない。リリース側で一方的に完結する
  手段（破壊的変更の明示と対処方法の提示）だけがスケールする。
- スキル（`.claude/skills/`）は下流リポジトリでカスタマイズされる前提のため、
  「新版ファイルの再コピー」は普遍的な対処方法にならない。対処方法の主語はファイルではなく
  **契約**でなければならない。

## 決定

1. **後方互換レイヤを書かない。** 旧フォーマット読み取り・フォールバック・バージョン分岐を
   実装しない。レビュー（人間・agent とも）は互換フォールバックの追加を指摘・要求しない。
2. **破壊的変更は CHANGELOG / GitHub Release notes の BREAKING セクションで明示し**、
   次の 3 要素を必ず記載する:
   - **壊れる契約**: 何を前提にしていた何が動かなくなるか
   - **影響の判定方法**: 下流 repo が影響を受けるかを確認する手段（grep 等の 1 コマンドが理想）
   - **適用指針**: 未カスタマイズなら該当ファイルの再コピーで可の旨。カスタマイズ済み repo
     向けには契約変更点の説明と上流 commit / PR への参照（diff を見て自分の版に移植できるように）
3. **（関連原則）スキルを跨ぐ契約は SKILL.md の散文ではなく CLI / harness 層（コード）に置く。**
   下流のスキルカスタマイズが契約を壊しにくくなり、BREAKING の適用指針も
   「呼び出し 1 行の追加・差し替え」で説明が完結する。

## 帰結

- 移行コード・互換フォールバックのレビューと保守が消える。
- 破壊的変更の伝達は release notes 側の責務になる。`/release` skill の Step 3（CHANGELOG 生成）が
  BREAKING エントリの 3 要素充足を確認する。
- 曖昧な状態で破壊的操作を行わない fail-safe（例: 検出不能時は上書きせず ABORT）は
  後方互換ではなく恒久的な安全設計であり、本 ADR の対象外（引き続き推奨される）。
- 外部公開 API の提供など前提が変わった場合は本 ADR を改訂する。
